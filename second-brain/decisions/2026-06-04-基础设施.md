# 2026-06-04 - 基础设施与工作流

### 决策
营销网站 + 支持页面合并，使用 GitHub Pages 静态部署，每个 App 独立隐私/条款页。

### 原因
独立开发者不需要维护多个网站，GitHub Pages 免费、稳定。每个 App 有独立 URL（/privacy/appname、/terms/appname）可满足 App Store / Google Play 审核要求。

### 放弃方案
多个独立网站，或使用 CMS/后端。

### 放弃原因
维护成本高，GitHub Pages 静态方案完全满足需求。

---

### 决策
AI 工作流分级：小改动直接手改，中等任务用 Claude Code，大任务用 heavy_task.skill（OpenCode + DeepSeek 执行）。

### 原因
把所有改动都走完整工作流成本太高（改个 padding 要走 5 步流程）。bug 修复最怕低质量模型无限试错导致 bug 扩散，大型重复劳动才适合低成本模型批量执行。

### 放弃方案
所有任务统一走 Agent 工作流。

### 放弃原因
工作流本身也是成本，过度流程化会让小改动效率比手改还低。

---

### 决策
UI 还原工作流：截图视觉理解 + 南湖 MCP 获取 OC 代码作为中间层，AI 按 Flutter 习惯重构。

### 原因
Figma MCP 原始节点树噪音多，AI 理解困难；OC/UIKit 代码已经工程化，UIView→Container、UILabel→Text 映射直接，AI 转 Flutter 更稳定。

### 放弃方案
直接用 Figma MCP 完整节点树生成 Flutter 代码。

### 放弃原因
Figma 节点树噪音多（无意义 Frame/Mask/Vector），AI 容易生成结构错误的代码，后期维护困难。

---

### 决策
服务器安全组只开放 22/80/443/8443 端口，其他端口不暴露公网。

### 原因
公网服务器每天都有自动扫描，暴露 SSH/MySQL/Redis 等端口极易被攻击。22/80/443 是必要端口，8443 用于 sing-box 代理。

### 放弃方案
0.0.0.0/0 全部端口允许。

### 放弃原因
安全风险极高，生产服务器被扫描入侵概率极大。

---

### 决策
TLS 兼容性修复：在 Nginx ssl 配置中加 `ssl_conf_command Groups X25519:secp256r1:secp384r1` 禁用 ML-KEM。

### 原因
Nginx 1.31.1 + OpenSSL 3.5 默认启用后量子密钥交换（ML-KEM），导致用域名访问时 TLS 握手被 reset，IP 直连正常。

### 放弃方案
降级 Nginx/OpenSSL。

### 放弃原因
升级版本解决安全问题，降级是退步；用配置禁用 PQC Groups 是最小侵入性修复。
