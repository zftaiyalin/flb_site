# Lessons

## 产品

- MVP 阶段的核心目标是验证"核心循环是否成立"，不是验证收入
- 独立开发前期平庸是正常阶段，增长是阶梯式的（长期没变化 → 某天突然跨台阶）
- 社区需要内容密度，独立开发前期用户量不足，过早社区化容易死
- RevenueCat 跑通一次后，新项目基本只需复制配置 + 换 product id，主要成本在平台配置
- "支付基础设施"只要第一次跑通，后面新 App 复用成本极低；真正浪费时间的是 App Store/Google Play 的平台配置
- 产品真正有价值的不是代码，而是产品判断、商业判断和决策逻辑，这些几年后还值钱
- 独立开发最危险的模式：AI 降低了"开始一个项目"的成本，导致永远在新项目新鲜感里，旧项目长不大
- 技术能力 ≠ 长期商业成功；真正难的是长期更新、用户增长、SEO/ASO 长期积累、商业化优化

## 技术

- 阿里云安全组必须只开放必要端口：22/80/443，不要全端口开放，公网服务器每天都被扫描
- Nginx 1.31.1 + OpenSSL 3.5 有 ML-KEM 后量子密钥交换兼容问题：症状是域名访问 TLS reset，IP 直连正常。修复：`ssl_conf_command Groups X25519:secp256r1:secp384r1`
- Flutter 白噪音 Android 必须处理：后台播放 / 锁屏播放 / 通知栏控制 / 电池优化 / 息屏不断音 / 音频焦点，这些需要原生 Kotlin 插件
- Firebase MVP 阶段只接 Core + Analytics + Crashlytics，不做复杂事件埋点，先跑起来
- AdMob ARPU 指标需要 Firebase 关联才能计算（Firebase 提供 DAU 数据，AdMob 只有收入数据）
- Google Play 内购测试：添加 License Tester 账号后，支付弹窗会出现"Test card always approves"，完全不需要真实扣费；订阅时间也会被压缩（月付 → 几分钟）
- 宝塔适合 MVP 阶段快速部署（Nginx/Docker/SSL 可视化），但不适合大型生产环境；建议把 sing-box 用 systemd 而非宝塔 Docker 运行
- OpenCode + DeepSeek 适合重复劳动/样板代码，不适合 Flutter 复杂状态/动画/生命周期 bug（低质量模型修 bug 容易扩散）

## AI 工作流

- App Store 宣传图生成应该用"模板替换"思路而非"重新生成"：先生成一张母版，后续只替换截图和文案，Figma 做排版系统，AI 做美术素材，稳定性高很多
- 关键 Prompt：`保持第一张图中除手机截图区域和宣传文案区域之外的所有视觉元素完全不变，不要重新设计海报，只替换截图和文案` — 这个"模板继承 Prompt"比描述风格的 Prompt 可复用性高很多
- 蓝湖/南湖 OC/UIKit 代码是比 Figma 节点树更好的 Flutter 重构中间层：OC 代码已工程化，UIView→Container 映射直接，AI 理解更准确
- JSX 是 AI 最容易理解的 UI 中间语言，JSX→Flutter 比设计稿→Flutter 稳定，因为 JSX 已进入"组件树语义层"
- AI 任务分级：微小改动自己手改最快；中等任务 Claude Code/Codex；大任务/重复劳动才用 heavy_task.skill（OpenCode+DeepSeek）
- DeepSeek 适合：重复劳动、样板代码、批量 CRUD、DTO 生成；不适合：架构设计、复杂状态、动画、高质量 bug 修复
- `heavy_task.skill` 的核心价值：将 Claude Code 定位为"分析派单验收"，OpenCode+DeepSeek 定位为"执行层"，每任务最多改 3 个文件，失败最多重试 2 次
- AI 生成图片选择：ChatGPT 网页版生图不消耗 Codex 额度，两套配额体系独立；大量生图放 ChatGPT 网页版，编程放 Claude Code/Codex
- Codex 生图（GPT Image 2）会消耗 3~5 倍 Codex 额度，MVP 阶段不在 Codex 里生图

## 上架审核

- App Transfer（App 所有权转移）必填项：Support URL / Privacy Policy URL / App Review Contact（姓名/电话/邮箱），缺任何一个"批准"按钮都不可点击
- 电话格式：`+86 手机号`（国际格式，苹果审核可能真的打电话确认）
- AdMob 验证（iOS）：App 必须在 App Store Connect 创建 App record（即使未上架），AdMob 才能验证 App 所有权；播放广告 ≠ 立即通过验证
- AdMob 账号不活跃提示：播放广告后台有延迟（几小时到一天），只要广告请求到达后台就能续命，不要立刻判断无效

## 广告变现

- 国内用户：优先穿山甲（CSJ）；海外用户：优先 AdMob；MVP 阶段不上 MAX 聚合（配置复杂，包体大）
- RevenueCat Webhook 必处理事件：INITIAL_PURCHASE / RENEWAL / EXPIRATION / CANCELLATION；BILLING_ISSUE 不要立即封权限（可能在宽限期）
- 广告设计原则：不能高频打断、不破坏沉浸感；最好的广告位是"高情绪节点"（如游戏结局/重要奖励）
- Flutter 广告层应该抽象成接口（AdService），国内 CSJ 实现和海外 AdMob 实现分开，方便切换
